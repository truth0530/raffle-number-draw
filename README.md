# 현장 추첨 웹앱 (Raffle 2026)

행사장에서 쓰는 추첨 웹앱입니다. **링크 하나(도메인 루트 `/`)** 에서 세 모드를 선택합니다.

1. **번호표 추첨** (`/numbers/*`) — 종이 번호표(1~N)를 나눠준 뒤 화면에 랜덤 번호를 뽑아
   띄우는 방식. 서버·DB 불필요(순수 클라이언트, `localStorage` + `BroadcastChannel`).
2. **유리병(QR) 응모 추첨** (`/stage`·`/enter`·`/control`) — 관중이 QR로 이름+전화 뒤4자리를
   응모하면, 무대 화면의 유리병 안에 이름 버블이 실시간으로 쌓이고, 마감 후 병을 뒤집어
   탈락자를 배출해 당첨자만 남기는 물리 연출 추첨. (Supabase Postgres + Vercel)
3. **시나리오 테스트** (`/test/*`) — 도우미 연습용 샌드박스. 8자리 테스트 코드
   (`NEXT_PUBLIC_TEST_CODE`, 기본 `RAFFLE26`)로 진입하며, ②와 **같은 화면 컴포넌트**를
   브라우저 로컬 엔진(`simRaffle.ts`)에 연결해 전체 시나리오(응모→마감→추첨→추가추첨→리셋)를
   연습한다. 데이터가 브라우저 안에만 있어 **실제 행사에 어떤 영향도 없음**.

## 유지보수 일관성 (한 곳 고치면 셋 다 반영)
- 무대/리모컨/응모 화면: `StageView`·`ControlView`·`EnterView` 공용 컴포넌트 — 실제(`live`)와
  테스트(`test`)는 전송 계층(mode prop)만 다르다.
- 씬 상태머신: `src/lib/scenes.ts` 단일 진실 — 서버 API와 테스트 샌드박스가 공유.
- 가상 이름 생성: `src/lib/koreanNames.ts` — 서버 리허설(/api/rehearsal)과 샌드박스가 공유.

## 화면

### 유리병(QR) 응모 추첨 — 관리자 진입점은 `/control` 하나
- `/control` — **관리자 리모컨(휴대폰/노트북)**. 관리자가 유일하게 접속하는 주소.
  무대(프로젝터) 창 열기(상단 **[무대 화면 ↗]**), 무대 표시 프리셋(**QR만 크게 /
  QR+항아리 / 항아리만**), 씬 전이·마감·추첨·추가추첨·병 제어·리셋 전부 여기서.
  토큰은 `#token=...` 해시로 1회 주입되어 localStorage에만 저장됩니다.
- `/stage` — **무대(프로젝터)**. 리모컨에서 열어주는 표출 전용 창. 직접 접속해도 됨.
- `/enter` — **관중 응모(QR 목적지)**. 관중은 무대 QR로만 들어온다(주소 노출 불필요).

### 번호표 추첨
- `/numbers/admin` — 발표자(관리자). 범위·개수·속도 설정, 추첨/채우기, 번호 상태 표시.
- `/numbers/show` — 슬라이드쇼(프로젝터). 관리자에서 **[슬라이드쇼 창 열기]** 로 띄웁니다.

## 유리병 추첨 운영 순서
0. `/control` 접속(토큰) → **[무대 화면 ↗]** 로 프로젝터 창을 열고, **무대 표시**
   프리셋으로 QR만/QR+항아리/항아리만을 선택.
1. `/control` 에서 **응모 현황 화면으로** → 관중 응모(QR/COLLECTING).
2. **응모 마감**(FROZEN) → 이후 응모는 409로 차단.
3. **추첨 시작 (20명)** → 서버가 트랜잭션으로 당첨 확정, 병이 뒤집힘(DRAWING).
4. **코르크 열기** → 설정한 소요시간 동안 탈락 버블이 주둥이로 배출.
5. **당첨자 명단 공개**(WINNERS) → 필요 시 **추가 추첨**(기존 당첨자 자동 제외).
6. 행사 후 **전체 리셋** → 스냅샷 JSON이 관리자 브라우저에 자동 다운로드된 뒤 초기화.

### 리허설 (프로덕션 시연)
`/control` 의 **리허설(가상 응모)** 섹션에서 20/100/300명 등 가상 인원을 투입하면
실제 관중 없이 전체 시나리오(유입 연출→마감→흔들기→추첨→추가추첨)를 시연할 수 있습니다.
가상 응모는 `ip="rehearsal"` 로 표식되어 **가상 응모만 삭제**로 선별 제거 가능
(단, 추첨 시작 후에는 전체 리셋으로만). 리허설이 끝나면 **전체 리셋** 후 본행사를 시작하세요.

## 설계 요점
- **씬 상태머신**: `QR → COLLECTING → FROZEN → DRAWING ↔ WINNERS` 전이만 서버가 허용.
- **당첨 확정은 서버 트랜잭션**: `Winner.entryId unique` 로 추가추첨 중복이 원천 차단.
  셔플은 CSPRNG(`crypto.randomInt`) Fisher–Yates.
- **개인정보 최소 노출**: 공개 API(`/api/entries`)는 id+이름만 반환. 뒤4자리는
  당첨 공개 씬(DRAWING/WINNERS)의 `/api/state` 명단에만 포함.
- **물리 연출은 클라이언트 전용**: 병은 물리세계에서 영원히 고정, 기울기는 중력벡터
  회전+렌더 회전으로만 표현(터널링·공중부양 구조적 차단). 배출은 스크립트(pluck).
  당첨/탈락 결과는 항상 서버 데이터가 진실.

## 개발
```bash
npm install
cp .env.example .env   # DATABASE_URL(6543)/DIRECT_URL(5432)/ADMIN_TOKEN 설정
npm run db:push        # raffle 스키마에 테이블 생성
npm run dev            # http://localhost:3000
npm run gate           # e2e 검증 게이트(dev 서버 실행 후) — 응모→마감→추첨→리셋 전체
```
- 스택: Next.js 16 + React 19 + TypeScript + Prisma + Supabase Postgres + matter-js.
- 배포: Vercel (GitHub 푸시 → 자동 배포). `DATABASE_URL` 은 **세션 풀러(5432)** 사용
  — 트랜잭션 풀러(6543)는 Prisma 인터랙티브 트랜잭션을 끊는다(P2028 실측).
- 주의: 로컬 개발과 프로덕션이 같은 Supabase DB를 공유하므로, 행사 직전 관리자에서
  **전체 리셋**을 한 번 누르고 시작할 것. `npm run gate` 도 데이터를 리셋한다.
